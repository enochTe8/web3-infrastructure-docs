# EigenLayer AVS Architecture

EigenLayer enables a new class of decentralized services called **Actively Validated Services (AVSs)**.

AVSs are external protocols that leverage Ethereum's validator security through EigenLayer’s restaking mechanism.

Instead of building independent validator networks, these services use restaked ETH to secure their operations.

---

## Overview

An Actively Validated Service is a protocol that requires decentralized validation to function securely.

Examples include:

• decentralized oracle networks  
• data availability layers  
• cross-chain bridges  
• rollup sequencers  
• off-chain computation verification systems  

Through EigenLayer, these services can access Ethereum’s validator set by allowing validators to **opt-in and perform additional tasks**.

---

## Core Architecture

The AVS system consists of several interacting layers:
Ethereum Consensus Layer
│
│
EigenLayer Restaking Layer
│
│
Operator Infrastructure
│
│
Actively Validated Services (AVSs)

Each layer plays a different role in extending Ethereum's security.

---

## Key Components

### Restakers

Restakers are ETH holders who delegate their staked ETH to EigenLayer.

By opting into restaking, they agree that their stake may be **slashed** if the operator they support misbehaves while validating AVS tasks.

---

### Operators

Operators run the infrastructure responsible for executing validation tasks.

They may perform duties such as:

• verifying oracle data  
• validating cross-chain messages  
• checking rollup state transitions  
• confirming data availability

Operators receive delegated restaked ETH from restakers and use it to participate in AVS validation.

---

### Actively Validated Services (AVSs)

AVSs are the protocols that require validation.

Each AVS defines its own:

• validation rules  
• reward mechanisms  
• slashing conditions  
• operator requirements  

This allows multiple independent services to share the same validator security pool.

---

## Validation Workflow

A simplified validation flow typically follows this process:

1. A service request is generated by the AVS.
2. Operators assigned to the AVS perform validation tasks.
3. Results are submitted back to the AVS protocol.
4. Rewards are distributed to participating operators and restakers.
5. Misbehavior can trigger slashing penalties.

This creates a **cryptoeconomic incentive system** for maintaining correct behavior.

---

## Security Model

The security of AVSs relies on economic incentives.

Operators and restakers risk losing part of their staked ETH if they violate protocol rules.

This aligns validator incentives with the integrity of the services they secure.

However, because validators may secure multiple AVSs simultaneously, the system introduces **interconnected security risks**.

Failures in one service could potentially affect participants across multiple protocols.

---

## Benefits of the AVS Model

EigenLayer’s AVS architecture provides several advantages:

• rapid security bootstrapping for new protocols  
• shared validator infrastructure  
• reduced cost of launching decentralized services  
• stronger economic security through restaked ETH

These properties make AVSs an attractive foundation for building new Web3 infrastructure layers.

---

## Challenges and Risks

Despite its advantages, the AVS model introduces several technical challenges.

These include:

• complex slashing conditions across multiple services  
• validator centralization risks  
• coordination between independent protocols  
• increased systemic risk within the shared security network

Careful protocol design is required to mitigate these issues.

---

## Future Outlook

If widely adopted, AVSs could become a core infrastructure layer in the Ethereum ecosystem.

Rather than every protocol building independent validator networks, many services may rely on **shared cryptoeconomic security** provided by EigenLayer.

This could accelerate the development of decentralized infrastructure and expand the capabilities of Ethereum-based systems.
